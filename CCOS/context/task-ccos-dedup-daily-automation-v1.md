## 需求概述

在 `ccos` 节点内完成一轮去冗余改造，并落地“AI 生成 + 人类审核 + 自动上报公司日报”的最小闭环能力，解决低消费输出和认知复利不足问题。

## 验收标准

1. 去冗余治理文档已沉淀并纳入知识索引。
2. Hub 与 Node 双层任务线均已登记该任务并可路由。
3. 明确自动日报流程：输入、生成、审核、上报、反馈五段闭环。
4. 定义可量化指标：状态漂移率、日报消费率、审核时长、证据完整率。

## 风险表

| 风险 | 影响 | 缓解方案 |
|---|---|---|
| 自动生成内容偏离事实 | 上报失真、知识污染 | 人审门禁强制“事实校验 + 证据链接” |
| 审核负担过高 | 机制无法持续执行 | 严控模板字段，审核动作压缩为 3 步 |
| Hub/Node 状态继续漂移 | 任务路由失真 | 增加一致性 lint 与单源策略 |

## 待确认问题

1. 公司日报上报通道是 API 还是固定文档平台复制粘贴。
2. 审核角色是否固定为开发者本人，还是可委托他人。
3. 日报必须字段是否有企业模板约束（例如工时、风险、次日计划）。

## 建议执行顺序

1. 先完成 `T4` 字段契约冻结。
2. 同步推进 `T5/T6` 并做一次端到端演练。
3. 以 1 周为周期运行 `T7`，再根据指标做 V1.1 收敛。

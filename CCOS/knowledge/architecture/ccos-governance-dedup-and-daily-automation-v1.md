# CCOS 去冗余治理与自动日报闭环（V1）

## 1. 目标

1. 降低 CCOS 在多项目协作中的重复维护成本。
2. 将日报从“流水输出”升级为“决策驱动闭环”。
3. 在保持 AI 自动化效率的同时，保留人类审核的治理门禁。

## 2. 产品定位

CCOS 不是通用 AI 助手，而是 AI 协作系统的控制平面：

1. 定义任务路由与执行边界（`project_id/repo_root/ccos_node`）。
2. 定义并发协作的认领、锁和冲突治理。
3. 定义知识资产的单源、索引和验证门禁。

与 AI 工具分工：

1. AI 工具负责“生成内容、执行动作”。
2. CCOS 负责“状态约束、门禁校验、可追溯治理”。

## 3. 必要性判定

以下条件满足 2 条及以上时，CCOS 为高必要：

1. 多项目并行，且任务跨仓切换频繁。
2. 多 AI 或多人并发推进，存在冲突风险。
3. 任务周期长于单会话，要求跨天可接手。
4. 组织有合规要求（日报、审计、过程留痕）。

以下条件成立时，CCOS 可简化为轻量模式：

1. 单人短链任务，低并发低风险。
2. 无需跨工具迁移和审计追踪。

## 4. 去冗余基线（V1）

### 4.1 单源原则

1. 任务核心状态单源：项目 Node `CCOS/context/task-index.md`。
2. Hub 任务线索引为路由投影，不重复维护核心状态字段。
3. 知识正文单源：`CCOS/knowledge/**`；会话文档仅保留过程与证据。

### 4.2 规则执行化

1. `MUST` 规则必须可机器校验，无法校验的降级为 `SHOULD`。
2. 新增规则时必须绑定对应检查器或 lint 规则。
3. 优先维护少量高价值规则，避免“规则多但不可执行”。

### 4.3 输出消费化

1. 每份自动输出必须声明 `consumer`（消费人/系统）。
2. 每份自动输出必须声明 `decision_needed`（需要的动作）。
3. 无消费方的输出不入长期知识层。

## 5. 自动日报闭环（AI 分析 + 人审）

## 5.1 输入层

1. Git 变更（commit、diff、影响文件域）。
2. 任务状态（进行中、阻塞、待确认）。
3. 当日决策与风险记录（来自 task/context）。

### 5.1.1 日报口径分层

1. `证据投影日报`：默认模式，目的是稳定映射 Git / taskline 投影，便于自动化、审计与跨项目聚合。
2. `实际工作日报`：仅在用户明确要求“今日实际工作 / 包含未提交改动”时启用，允许跨仓聚合相关项目的真实提交、当日 journal 和必要 diff。
3. 两种口径不得混名输出；若采用“实际工作日报”，必须在生成链路里显式说明口径。

### 5.1.2 仓级范围对齐

1. Hub 仓 `ccos` 中的 `docs(taskline)` 提交通常只是项目仓真实工作的投影，不应被误判为全部工作事实。
2. 若用户要求“实际工作日报”，必须优先按 taskline `repo_root` 或联邦注册表定位相关项目仓，再读取对应仓的提交 / journal / 必要 diff。
3. 只有在确认项目仓没有额外工作证据时，才允许把 Hub 投影提交直接当成主要事实来源。

### 5.1.3 Diff 提取安全边界

1. `Diff-Fallback` 只允许从文档类文件提取 `DONE/TODO/NOW/WAITING/工时`。
2. 代码文件中的 `done/todo` 变量、数组初始化或注释不得被当成日报语义。
3. 若无法从文档类文件提取稳定语义，应退回“文件级摘要 + 人审”模式，而不是强行生成伪语义条目。

## 5.2 AI 生成层（四类产物）

1. 企业日报草稿（标准模板）。
2. 风险与阻塞清单（按严重级别）。
3. 可沉淀知识候选（含证据链接）。
4. 次日计划草案（含依赖与优先级）。

## 5.3 人审门禁层

审核动作只保留三项：

1. 事实正确性确认。
2. 是否触发知识沉淀确认。
3. 是否触发任务状态变更确认。

## 5.4 发布层

1. 企业日报自动上报（审核通过后）。
2. CCOS 仅落盘“通过审核”的结构化结果。
3. 驳回项进入 `review_feedback`，用于下轮提示优化。

## 5.5 反馈学习层

1. 记录“被采纳/被驳回”原因。
2. 每周回顾提示词与规则效果。
3. 对高误报项进行规则降噪。

## 6. 防止“自动化不提升认知”的约束

1. 每日日报至少包含 1 条“决策-依据-风险”三元组。
2. 每周至少 1 条“被推翻结论及原因”复盘项。
3. 无证据引用的知识候选禁止自动入库。
4. 审核耗时超过阈值（建议 8 分钟）必须优化流程而非增加模板。

## 7. 最小可用能力（MVP）

1. 任务三元路由和状态单源。
2. Hub/Node 一致性 lint。
3. 自动日报草稿 + 人审 + 自动上报。
4. 知识候选入库门禁（证据必填）。

## 8. 验收指标（V1）

1. 状态漂移率：Hub 与 Node 不一致任务线比例 < 5%。
2. 日报消费率：日报产生的明确决策项比例 > 70%。
3. 审核效率：单日报审核时长 3-8 分钟。
4. 知识质量：入库候选中具备证据链接比例 = 100%。

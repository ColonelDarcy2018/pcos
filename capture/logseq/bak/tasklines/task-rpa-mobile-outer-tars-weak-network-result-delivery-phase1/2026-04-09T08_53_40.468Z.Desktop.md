## 背景

该任务线承接两条已有主线之后的“下一轮高价值优化”：
1. `tars-commander-retry-hardening` 已完成稳定 `requestUuid`、同步/异步/持久化补偿、process restart recovery 等基础兜底。
2. `tars-diagnostic-logging-mvp` 已完成诊断包、离线分析和平台态观测增强，能够更稳定地把“客户端未获确认”“系统级弱网”“平台侧更可疑”分层。

但最近两份客户现场诊断包表明，当前塔斯 App 仍会在系统级弱网场景下出现“平台显示丢失结果 / 结果未稳定确认”的投诉。现有代码已不是“完全没有重试”，而是“仍缺少弱网下的实时稳态策略、关键确认信号与更强的队列整形能力”。

本任务线用于沉淀后续 Android 侧仍值得继续投入的一轮优化，避免后续 AI 在“要不要继续加重试”“是不是已经无优化空间”“为什么还需要客户优化网络”三个问题上反复回溯同一批事实。

## 当前同步结论（摘要）

1. 当前塔斯 App 已具备基础弱网兜底：
   - `report` 同步最多 `3` 次，等待约 `21s`
   - 异步内存重试最多 `3` 次
   - 持久化补偿按 `2s/4s/6s...` 退避，封顶 `60s`
   - `requestUuid` 已按稳定身份复用，避免同一上报事件每次重试都生成新主键。
2. 最近客户现场诊断包 `diagnostics_UK7DXGJJHY9XFY9P_20260409_155052.zip` 已证明：
   - 在已观测的 `113.148s` 窗口里，至少有 `18~21s` 的系统级严重弱网
   - 同一 `workUuid/workExecuteUuid` 的 `RUNNING` 上报有 `93` 次成功、`1` 次失败
   - 失败点为 `2026-04-09 15:50:50.488`，`1.957s` 后恢复成功
   - 同窗系统日志存在 `NETWORK_ABNORMAL` 与 `tcp fail rate 89~100`
   - 这支持“设备网络抖动主因”，不支持“塔斯平台域名整体不可达”或“App/流程自身已观测到异常”。
3. 当前投诉里的“丢失结果”更准确地说，是“平台未在自己的确认窗口内稳定拿到关键上报回执”；后续补发可以提高最终一致性，但不能保证平台过程态一定不报警。
4. App 仍有一轮高价值优化空间，但方向不再是“盲目加长重试时间”，而是：
   - 降低重复在途与弱网放大效应
   - 优先保护关键事件上报
   - 提升客户端/平台边界定责能力
5. 即便完成本任务线，仍不能承诺“系统级严重弱网下实时稳定一定成立”；用户侧网络优化依旧是必要条件，不是可选项。

## 为什么还要继续做 App 优化

1. 当前已有补偿，但仍缺“实时稳态”：
   - 长补偿能提高最终补发概率，不等于平台过程态不会先显示“丢失结果”。
2. 当前 `RUNNING` 心跳是 `1s` 一轮，与 commander 主循环共享节奏；弱网时仍可能把相同语义的上报重复压入队列。
3. 当前诊断已能证明“网络主因”，但仍缺“客户端已确认发送 / 平台已确认接收 / 最终由谁丢失”的硬边界字段。
4. 继续做一轮客户端优化，价值在于：
   - 降低误报与放大
   - 改善排障效率
   - 让后续 AI 和客户更快区分“可由 App 收敛”与“必须做网络整改”

## 本任务线建议范围

### A. 结果上报稳态治理

1. 对 `RUNNING` 心跳做“同 stable key 队列去重 / 在途合并”，避免弱网时同一语义重复进入 dispatch。
2. 将 `FINAL_STATUS`、`FIRST_RUNNING`、`RUNNING_HEARTBEAT`、`PROCESS_RESTART_RECOVERY` 分 source 管理优先级：
   - `FINAL_STATUS` 最高
   - `FIRST_RUNNING` 次高
   - `RUNNING_HEARTBEAT` 可在弱网时降频或合并。
3. 引入“同 key 在途保护”：
   - 同一 `workStatus + reason + workExecuteUuid` 在请求未确认时，不再无限创建新的同类上报。

### B. 弱网自适应策略

1. 新增轻量 `WeakNetworkState`：
   - 基于连续 timeout / retryable 网络错误 / call elapsed 过长判定
   - 状态至少区分 `NORMAL / DEGRADED / SEVERE`
2. 在 `DEGRADED/SEVERE` 下：
   - `RUNNING` 心跳从 `1s` 降到 `3~5s`
   - 若同类上报已有 pending / in-flight，则只刷新最后活跃时间，不重复发包
   - `FINAL_STATUS` 保持高优先级，不受降频影响
3. 为弱网模式增加结构化 trace，便于诊断包直接回答“App 当时是否进入弱网降级模式”。

### C. 客户端确认边界增强

1. `/api/platform_commander_state` 新增结果上报成功信号：
   - `lastReportSuccessAt`
   - `lastReportAckRequestUuid`
   - `lastReportAckWorkStatus`
   - `lastReportAckCode`
   - `lastReportSource`
2. 新增在途与补偿摘要：
   - `inFlightReportKey`
   - `inFlightReportSinceMs`
   - `weakNetMode`
   - `consecutiveRetryableReportFailures`
3. 让诊断包和离线分析器能直接输出：
   - 弱网窗口
   - 失败到恢复时间
   - 是否命中过弱网降级
   - 客户端最后一次拿到 ACK 的时间点

### D. 长时间网络体检能力

1. 在现有诊断脚本基础上新增“长时采样模式”：
   - 目标是连续 `30~60 分钟` 抓取系统 QoE / commander trace / report trace
2. 离线分析新增每小时统计：
   - 可观测时长
   - 严重弱网时长
   - `NETWORK_ABNORMAL` 次数
   - report fail / success 比例
3. 该能力用于“证明网络整改必要性”，不是当前 App 主链逻辑本身的一部分。

## 非目标

1. 不承诺通过客户端代码让系统级严重弱网场景下的实时上报 `100%` 稳定。
2. 不把“长补偿”误包装成“平台永远不会显示丢失结果”。
3. 不在本任务线里改业务流程脚本、微信选择器或小鹏业务规则。
4. 不把平台服务端幂等、确认窗口和晚到补发逻辑直接算作客户端单侧责任。

## 与现有任务线的关系

1. `tars-commander-retry-hardening`
   - 已完成基础补偿和 `requestUuid` 稳定化
   - 本任务线不是推翻，而是在其上继续做弱网稳态与可观测性升级
2. `tars-diagnostic-logging-mvp`

   - 仍承接更广义的 runtime 状态机与 commander 轮询事实
   - 本任务线聚焦“弱网 + 上报 + 丢失结果”收口，不再把范围摊大

## 验收标准

1. 相同 `RUNNING` stable key 在任意时刻最多只有 `1` 个 in-flight / `1` 个 pending 语义，不再无限堆同类上报。
2. `FINAL_STATUS` 在弱网场景下具备显式优先级，不会被普通心跳淹没。
3. `platform_commander_state` 能直接回答“客户端最近一次拿到平台 ACK 是什么时候、哪种上报、哪个 requestUuid”。
4. 诊断包和离线分析器能输出“已观测窗口内的弱网持续时长、失败到恢复时间、当前只是弱网还是明确断网证据不足”。
5. 对外结论模板能清楚区分：
   - App 还能继续收敛的范围
   - 只能靠用户网络整改解决的范围

# assets：人类认知资产层（Evidence → Patterns → Principles）

`assets/` 是 PCOS 的“人类表达空间”：允许自然语言、允许不完整、允许短期混乱。这里的目标不是把笔记分到正确的盒子里，而是让认知资产可以被长期积累、引用、复用与演化，并最终形成复利。

与之相对，任何“机器需要的结构”（索引、Schema、图谱、提示词、Agent 定义、生成物）都应进入 `machine/`，而不是反向污染 `assets/` 的书写自由。

---

## 三层认知抽象模型：不是分类，而是成熟度

这套结构的核心假设是：知识会随使用而“升维”，而不是一开始就能被正确设计出来。

- `assets/evidence/`（L1 Evidence）：事实与经验的沉淀
  - 原始案例、项目复盘、决策记录、实验与观察。
  - 允许强语境、允许主观、允许矛盾；关键是“可追溯、可复盘”。
- `assets/patterns/`（L2 Patterns）：可复用的方法与模式
  - 从多个 evidence 中抽出来的“稳定做法/结构/策略/技巧/流程”。
  - 关心可迁移性：适用边界、前提条件、失败信号、代价与权衡。
- `assets/principles/`（L3 Principles）：更稳定的原则与世界模型
  - 对 patterns 的再抽象：价值取向、方法论生成规则、系统哲学与世界观约束。
  - 关心长期一致性与反向指导：它应该能影响你下一次如何观察、如何做决策、如何取舍。

这三层不是“互斥分类”，而是一条上升路径：同一主题可以在不同阶段同时存在于多层，并以 Git 演进记录其变化轨迹。

---

## 自然演化：允许混乱与未完成

你不需要在写下第一句话时就知道它最终属于哪一层。

经验法则：

- 不确定放哪里：先放 `evidence/`。因为事实与语境永远有价值，过早抽象反而容易误导。
- 写作优先级：先把“当时发生了什么/我为什么这么判断”写清楚，再谈抽象与复用。
- 允许“暂存”：某些内容长期停留在 evidence 也没问题；系统的目标是复利，不是整洁。

---

## 目录是“认知阶段容器”，不是强约束模板

`evidence/ patterns/ principles/` 只提供一个“演化方向感”。

层内子目录（例如 `projects/ decisions/ methods/ models/ skills/ concepts/ philosophy/`）只是为了降低日常使用摩擦：它们可以新增、合并、迁移、废弃，不构成固定规范，更不要求字段模板。

如果未来你的工作流改变（工具、领域、团队协作方式变化），这三层仍应保持稳定；变化应主要发生在层内组织方式与 `machine/` 的自动化能力上。

---

## 与 machine 的边界：自由书写，后置复杂度

- `assets/`：给人看的、可读的、可长期维护的文本资产；允许模糊与演化。
- `machine/`：给机器看的、可计算的结构；允许复杂，但必须可验证、可重建、可回归。

当你想“让 AI/脚本更好地理解与利用这些资产”，优先选择在 `machine/` 增加索引与规则，而不是在 `assets/` 强行引入模板和固定字段。
